## 一、项目全貌

### 1.1 系统定位

诺保科销售拜访系统是一套面向医药销售团队的 **SaaS + 微信小程序** 解决方案。核心场景：销售外出拜访医院/医生时，通过小程序完成签到签出、记录拜访信息；管理员通过 Web 后台统一管理用户、医院基础数据、拜访记录和审批。

### 1.2 技术架构

```
┌──────────────────────┐     ┌──────────────────────┐
│   微信小程序          │     │   Web 管理后台        │
│   (销售/经理端)       │     │   (layuimini 框架)    │
│   9 个功能模块        │     │   ~10 个页面          │
└──────────┬───────────┘     └──────────┬───────────┘
           │                            │
           │  /api/*                    │  /admin/*
           ▼                            ▼
┌─────────────────────────────────────────────────┐
│              Laravel 8.x 后端                     │
│  ┌─────────────┐  ┌──────────────┐               │
│  │  api guard  │  │  admin guard │ ← 需改造隔离   │
│  │  (JWT)      │  │  (JWT/session)│               │
│  └─────────────┘  └──────────────┘               │
│                   共用 MySQL 数据库                │
└─────────────────────────────────────────────────┘
```

### 1.3 用户角色与职责

| 角色 | 使用端 | 核心职责 |
|:-----|:------:|:---------|
| 销售 | 小程序 | 完成每月拜访任务，签到签出，记录拜访信息 |
| 地区经理 | 小程序 | 完成拜访 + 查看手下员工进度 |
| 市场部 | 后台/小程序 | 会议后发起指定任务给销售 |
| SFE主管 | 后台/小程序 | 对预警销量下滑销售发起指定拜访 |
| 管理员 | Web后台 | 管理用户/医院/医生数据，查看全量拜访记录 |
| 超级管理员 | Web后台 | admin/NobelCRM，最高权限 |

## 四、优先级排序

### 排序原则

1. **安全先行**：权限隔离是所有后续工作的前提
2. **核心闭环优先**：管理员能管人 → 能看数据 → 小程序销售能用
3. **瓶颈依赖**：后端接口是前端页面的前置

### 第一优先级：P0 必选（核心交付基线）

| 顺序 | # | 内容 | 理由 |
|:----:|:--|:-----|:-----|
| 1 | B1-B4 | **权限分离改造** | 安全基础，不隔离则后台可被小程序用户访问 |
| 2 | A1 + A13 | **管理员登录** | 管理入口，后续所有管理功能的前置 |
| 3 | A2-A5 + A15 | **用户管理** | 管理员必须能管销售账号（需求文档核心诉求） |
| 4 | A6 + A16 | **拜访管理** | 核心数据查看，需求文档中拜访记录是管理后台重点 |
| 5 | C1-C5 | **小程序模块补全** | 让销售端功能完整可用，与后台开发可并行 |

**P0 合计**：约 6-7 天工作量

### 第二优先级：P1 推荐（增强管理能力）

| 顺序 | # | 内容 | 理由 |
|:----:|:--|:-----|:-----|
| 6 | A8-A11 | **医院管理 API** | 需求文档中医院管理是独立板块 |
| 7 | A17 | **审批管理界面** | 补签审核是签到闭环的关键环节 |
| 8 | A18 | **日报统计界面** | 经理查看团队数据 |
| 9 | A19 | **医院管理前端**（含医生子页面） | 对应后端 API |

**P0+P1 合计**：约 7.5-8.5 天

### 第三优先级：P2 可选（锦上添花）

| 顺序 | # | 内容 | 理由 |
|:----:|:--|:-----|:-----|
| 10 | A12 + A14 | **统计概览** | 控制台首页数据看板 |
| 11 | A7 | **强制修改拜访状态** | 紧急情况兜底 |
| 12 | A20 | **系统设置** | 打卡半径等配置 |
| 13 | — | Excel 批量导入/导出 | 需求文档强调但工作清单未列 |

**全量合计**：约 8.5-10 天

### 5.2 关键风险

| 风险 | 影响 | 缓解措施 |
|:-----|:----:|:---------|
| 需求文档中 Excel 导入/导出功能被遗漏 | 客户验收不通过 | 交付前与客户确认导入导出是否为必选项 |
| 小程序与后台使用同一数据库但不同 guard | 权限泄露 | B1-B4 改造必须严格测试，确保 admin guard 完全隔离 |
| 签到定位依赖高德地图 SDK | 小程序端集成复杂 | 确认高德 SDK 已集成，1km 范围判断逻辑在后端做校验 |
| 「拜访计划」标注暂不开发 | 后续可能新增需求 | 确认客户当前是否仍不开发，预留接口扩展点 |

## 六、附录：数据模型提示

根据需求文档，核心数据实体关系：

```
User (销售/经理/管理员)
 ├── role: 销售 | 经理 | 市场部 | SFE主管 | 管理员(role=9)
 ├── 上级领导 (自引用)
 └── 负责医院 (多对多)

Hospital
 ├── 编码/名称/地址/大类/小类/连锁分类/社会码
 ├── 大区/省份/城市
 └── Doctor[] (一对多)

Doctor
 ├── 姓名/科室/性别/职称/职能/手机/邮箱
 ├── Hospital (外键)
 └── Visit[] (一对多)

Visit (统一主表)
 ├── 类型: 客户拜访 | 手术跟台 | 会议举办 | 新开发医院
 ├── 日期/时间/工作选项/工作内容
 ├── 签到时间/签出时间/截屏图片
 ├── User (外键)
 ├── Hospital (外键)
 ├── Doctor (外键)
 └── 状态: 待签到 → 已签到 → 已签出 | 待补签

Task (派单任务)
 ├── 类型: 市场部任务 | SFE指定拜访
 ├── 发起人/接收人
 └── 状态: 待执行 → 已完成

Leave (休假)
 ├── 类型/起止时间/凭证
 └── User (外键)
```
